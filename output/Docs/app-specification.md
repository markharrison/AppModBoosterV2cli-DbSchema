## 1. Application Overview

### 1.1 Purpose

A modernised Expense Management System that enables employees to create, submit, and track expense claims, and managers to approve or reject expenses from their direct reports.

### 1.2 Application Name

Canonical prefix: **`expenses`**. All resources, projects, scripts, and workflows use this prefix.

## 4. Business Rules

### 4.1 Expense Lifecycle State Machine

```
(new) ‚îÄ‚îÄCreate‚îÄ‚îÄ‚ñ∂ Draft ‚îÄ‚îÄSubmit‚îÄ‚îÄ‚ñ∂ Submitted ‚îÄ‚îÄApprove‚îÄ‚îÄ‚ñ∂ Approved (final)
                    ‚ñ≤                            ‚îÇ
                    ‚îÇ                    Reject‚îÄ‚îÄ‚îÄ‚ñ∂ Rejected
                    ‚îÇ                                 ‚îÇ
                    ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ Edit & Resubmit ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
```

### 4.2 Status Transition Rules

| Current Status | Action          | New Status | Who Can Perform           | Side Effects |
|----------------|-----------------|------------|---------------------------|--------------|
| (new)          | Create          | Draft      | Any employee              | CreatedAt = UTC now |
| Draft          | Edit            | Draft      | Owner only                | ‚Äî |
| Draft          | Submit          | Submitted  | Owner only                | SubmittedAt = UTC now |
| Draft          | Delete          | (deleted)  | Owner only                | Hard delete |
| Submitted      | Approve         | Approved   | Manager of owner only     | ReviewedBy = manager's UserId, ReviewedAt = UTC now |
| Submitted      | Reject          | Rejected   | Manager of owner only     | ReviewedBy = manager's UserId, ReviewedAt = UTC now |
| Rejected       | Edit            | Draft/Rejected | Owner only            | ‚Äî |
| Rejected       | Submit          | Submitted  | Owner only                | SubmittedAt = UTC now (updated) |
| Approved       | (none)          | ‚Äî          | Final state               | No changes allowed |

### 4.3 Ownership Rules

- **Employees** can only see and manage their own expenses (where `Expense.UserId` = current user's UserId).
- Employees cannot see expenses belonging to other users.

### 4.4 Manager Scope

- **Manager's direct reports**: All Users where `User.ManagerId` = manager's UserId.
- Managers see submitted expenses only from their direct reports in the Pending Approvals view.
- Only the manager of the expense owner (not any manager) can approve/reject.
- Validation: `expense.User.ManagerId == managerId`.

### 4.5 Status Enforcement

- Status transitions must be enforced at the API level ‚Äî not via free-choice dropdown.
- The API must validate that the current status allows the requested transition before applying it.
- Invalid transitions return 400 Bad Request with a descriptive message.

### 4.6 Review Tracking

- On approve: set `ReviewedBy` = approving manager's UserId, `ReviewedAt` = DateTime.UtcNow.
- On reject: set `ReviewedBy` = rejecting manager's UserId, `ReviewedAt` = DateTime.UtcNow.
- These fields are set atomically with the status change.

### 4.7 Submission Tracking

- `SubmittedAt` is set to `DateTime.UtcNow` whenever the status transitions to Submitted.
- On resubmission (Rejected ‚Üí Submitted), `SubmittedAt` is updated to the new submission time.

### 4.8 Delete Restrictions

- Only Draft expenses can be deleted.
- Only the owner can delete.
- Approved and Rejected expenses cannot be deleted ‚Äî return 400.

### 4.9 Dual-Role Managers

- Managers can submit their own expenses (acting as employees).
- The UI shows both "My Expenses" and "Pending Approvals" views for managers.

### 4.10 Soft Deactivation

- `User.IsActive` and `ExpenseCategory.IsActive` support soft-deactivation.
- Inactive categories should be filtered from UI dropdowns but preserved for historical data.
- Inactive users should not appear in the user selector.

### 6.8 Confirmation Dialogs

Required for these actions:
- **Submit**: "Are you sure you want to submit this expense for approval?"

### 6.9 Amount Formatting

1. `appsettings.json` (base defaults)

## 9. Resilience

### 9.1 API App Startup Retry

- Database migrations and seeding run on API App startup.
- Retry logic: **5 attempts** with exponential backoff.
- Handles cloud cold-start timing where SQL Database or managed identity grant may not yet be available.

### 9.2 Health Endpoints

- `/live` ‚Äî Always returns 200 if the process is running and can respond to HTTP. No dependency checks.
- `/ready` ‚Äî Returns 200 only after migrations, seeding, and DB connectivity are confirmed. Returns 503 otherwise.
- Both endpoints are unauthenticated.
- Normal data requests return 503 Service Unavailable if the API is not yet ready.

### 9.3 Web App Resilience

- Web App must tolerate independent startup order (API may not be ready yet).
- When the API is unavailable or returns 503, the Web App shows a user-friendly "warming up" or "temporarily unavailable" message ‚Äî not a crash or error page.

## 13. Testing Strategy

### 13.1 Unit Tests (`Expenses.Api.Tests`)

- **Framework**: NUnit.
- **Scope**: Test all API controller actions, validation logic, and business rules.
- **Coverage targets**:
  - All GET endpoints return expected data.
  - POST endpoint creates expenses with correct defaults (Draft status).
  - PUT endpoints enforce status transition rules and ownership.
  - DELETE endpoint enforces Draft-only and ownership rules.
  - Invalid/missing required fields return 400 (not 500).
  - Status transition violations return appropriate errors.
  - Manager scope validation (only manager of owner can approve/reject).
- **Database**: Use in-memory or SQLite provider for fast isolated tests.

### 13.2 E2E Tests (`Expenses.Web.E2E`)

- **Framework**: Playwright with NUnit.
- **Scope**: Full end-to-end browser testing of the complete application.
- **Setup**: Spin up both API App and Web App as child processes in Development mode.

#### E2E Architecture Requirements

- **Sequential execution**: `[assembly: NonParallelizable]` ‚Äî no parallel test fixtures.
- **App health verification**: Each test `[SetUp]` verifies the Web App is responding. If a previous test crashed the app, fail fast with a clear message.
- **Navigation waits**: After form submissions (POST ‚Üí redirect ‚Üí GET), use `WaitForURLAsync` (not just `WaitForLoadStateAsync`) to ensure redirects complete before assertions.
- **Test data independence**: Tests that modify data (approve, reject, submit) must create their own test data ‚Äî do not depend on shared seed data that earlier tests may have modified.
- **Process cleanup**: `[OneTimeTearDown]` kills child processes with `entireProcessTree: true` to prevent port conflicts.

#### E2E Test Coverage

- Dashboard renders with correct data for both Employee and Manager views.
- My Expenses page lists expenses with correct status badges.
- Create Expense form submits successfully, new expense appears in list.
- Edit Expense form loads pre-populated data, saves successfully.
- Submit action changes status from Draft to Submitted (with confirmation).
- Approve action (as manager) changes Submitted to Approved.
- Reject action (as manager) changes Submitted to Rejected.
- Delete action removes Draft expense (with confirmation).
- User selector switches between users and updates visible data.
- Pending Approvals page shows only relevant expenses for the manager.

#### E2E Test Writing Rules

- Write tests AFTER inspecting the actual running application.
- Base all Playwright selectors, assertions, and expected text on what is actually rendered.
- Never write selectors against assumed HTML structure.
