## 1. Application Overview

The legacy application is an **Expense Management System** built on SQL Server. It allows employees to create and submit expense claims, and managers to approve or reject them.

- **Currency**: GBP (British Pounds)
- **Amount storage**: Minor units (pence) as `INT` to avoid floating-point rounding issues (e.g., £25.40 is stored as `2540`)
- **User context**: Simplified — no authentication/login; a UI user-selector switches between users
- **Database**: SQL Server (`dbo` schema), using `IDENTITY` columns, `DATETIME2`, `SYSUTCDATETIME()` defaults

## 4. Business Logic & Rules

### 4.1 User Roles and Permissions

| Role     | Permissions                                                        |
|----------|--------------------------------------------------------------------|
| Employee | Create, edit, submit, and view **own** expenses only               |
| Manager  | View and approve/reject submitted expenses from **direct reports**; may also submit own expenses |

- Managers can also be employees (dual role — they submit their own expenses too).
- Users see only data relevant to their role and user context.

### 4.2 Expense Lifecycle

```
(new) ──Create──▶ Draft ──Submit──▶ Submitted ──Approve──▶ Approved (final)
                                              ──Reject───▶ Rejected ──Edit & Resubmit──▶ Submitted
```

### 4.3 Status Transition Rules

| Current Status | Action           | New Status | Who Can Perform             |
|----------------|------------------|------------|-----------------------------|
| (new)          | Create           | Draft      | Employee                    |
| Draft          | Edit             | Draft      | Owner (Employee)            |
| Draft          | Submit           | Submitted  | Owner (Employee)            |
| Submitted      | Approve          | Approved   | Manager of the owner        |
| Submitted      | Reject           | Rejected   | Manager of the owner        |
| Rejected       | Edit & Resubmit  | Submitted  | Owner (Employee)            |
| Approved       | (none)           | —          | Final state — no changes    |

### 4.4 Key Business Rules

1. **Ownership**: Employees can only see and manage their own expenses.
2. **Manager scope**: Managers see expenses submitted by users where `Users.ManagerId = manager's UserId`.
3. **Edit restrictions**: Only Draft or Rejected expenses can be edited; only by the owner.
4. **Approval authority**: Only the manager of the expense owner can approve/reject — not any manager.
5. **Review tracking**: On approve/reject, set `ReviewedBy` = manager's UserId and `ReviewedAt` = current UTC.
6. **Submission tracking**: `SubmittedAt` is set when status changes to Submitted.
7. **Delete restrictions**: Approved and Rejected expenses cannot be deleted.
8. **Status enforcement**: Status must not be a free-choice dropdown — transitions must follow the lifecycle rules above.
9. **Confirmation**: Submit action requires user confirmation before executing.

## 7. Modernisation Considerations

### 7.1 Amount Handling
- Amounts are stored in **pence (minor units)** as `INT`. The modern UI must **divide by 100** for display (e.g., `2540` → `£25.40`) and **multiply by 100** when saving user input.
- Use proper currency formatting (`Intl.NumberFormat` or equivalent) for GBP display.

### 7.2 Status Transition Enforcement
- The legacy schema has no database-level enforcement of status transitions — it relies on application logic.
- The modern app **must enforce valid transitions** in the API/service layer (not just the UI).
- Status should not be a free-text or free-choice dropdown.

### 7.3 Review Audit Trail
- `ReviewedBy` and `ReviewedAt` must be set atomically when a manager approves or rejects.
- `SubmittedAt` must be set when transitioning to Submitted.
- The seed data has an inconsistency (Expense #2 is Approved but has no ReviewedBy/ReviewedAt) — the modern app should prevent this.

### 7.4 Self-Referencing Manager Relationship
- `Users.ManagerId → Users.UserId` creates a hierarchical relationship.
- The modern app needs to query "direct reports" efficiently (all users where `ManagerId = currentUser`).
- Consider: what happens if a manager is deactivated? The modern app should handle this gracefully.

### 7.5 User Authentication
- The legacy system uses a **user selector** (no auth). This is intentional for testing/demo purposes.
- The modern app should retain this simplified approach per the business requirements.

### 7.6 Receipt Files
- `ReceiptFile` stores a path string (e.g., `/receipts/alice/taxi_oct20.jpg`).
- The modern app needs to decide on file storage strategy (local filesystem, blob storage, etc.).
- Receipt upload is optional (nullable in schema).

### 7.7 Soft Deletes / Active Flags
- `Users.IsActive` and `ExpenseCategories.IsActive` support soft-deactivation.
- The modern app should filter inactive records from UI selectors but preserve them for historical data integrity.

### 7.8 Date/Time Handling
- Legacy uses `DATETIME2` with `SYSUTCDATETIME()` — all timestamps are UTC.
- `ExpenseDate` is `DATE` only (no time component).
- The modern app should maintain UTC storage and handle timezone display in the UI.

### 7.9 Delete Rules
- Approved and Rejected expenses cannot be deleted (business rule).
- Only Draft expenses should be deletable — the modern app should enforce this.

### 7.10 Dual-Role Managers
- Managers can submit their own expenses (acting as employees).
- The UI must support both the "My Expenses" and "Pending Approvals" views for managers.
